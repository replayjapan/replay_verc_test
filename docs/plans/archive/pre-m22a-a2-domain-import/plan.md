### Source data
`domain-content-remaining.json` at repo root contains 34 domain objects with full content: domainName, description, detailedDescription, valuePoints, potentialUses, category, richSummaryTitle, richSummaryIntro, useCasesTitle, useCasesSummary, useCases, searchExcerpt, searchKeywords, meta fields, and set membership data.

Content generated by Codex using the domain-content-template.json + content governance files. Blind-tested by Gemini, ChatGPT, and a separate Claude — all unanimously approved.

### 6 new domain categories to create
1. ファッション・スポーツ
2. 人材・ビジネス
3. ショッピング・EC
4. ダンス・パフォーマンス
5. アート・クリエイティブ
6. テクノロジー・教育

These supplement existing categories already seeded in A1.

### Domain sets to create/update
New members need to be added to existing sets, and any new sets defined in the JSON need to be created. Examples from the kickoff:
- Dance Set: dancer.jp, ダンサー.jp, salsa.jp
- Los Angeles Set: new members
- Singer Set: new members
- Others as defined in the JSON data

### Overlap handling
Some A1 featured domains may overlap with Codex output (e.g., 海外ホテル.com appeared in both). Rule: merge, don't duplicate. A1 content takes precedence for any fields that conflict — Codex content fills remaining fields only.

### Critical rule
**Do NOT rewrite Codex content.** Only fix:
- searchExcerpt over 80 double-byte char limit → truncate to limit
- Missing required fields → flag to Developer, do not invent
- Category name mismatches → match to seeded category names
- Registration date format issues → fix format
- Field structure mismatches → adapt to seed TypeScript format

If content quality issues are noticed, log them in STOP output for Developer awareness — do not edit the content.

## Requirements

### R1 — Read and validate JSON
Read `domain-content-remaining.json`. For each of 34 domains, validate:
- searchExcerpt within 80 double-byte char limit
- All required fields present (domainName, description, category, searchExcerpt at minimum)
- Category names match one of the seeded DomainCategory names (existing + 6 new)
- Registration dates are plausible real dates
- No duplicate domainNames with existing A1 seed domains (handle overlaps per context rules)
- Report: validation summary (pass count, fix count, flag count)

### R2 — Create 6 new domain categories
Add 6 new DomainCategory entries to the seed:
- ファッション・スポーツ
- 人材・ビジネス
- ショッピング・EC
- ダンス・パフォーマンス
- アート・クリエイティブ
- テクノロジー・教育

### R3 — Add 34 domains to seed
Wire each domain's JSON fields into seed TypeScript matching the existing A1 domain seed pattern. Map JSON fields to Payload collection fields. Handle rich text fields (detailedDescription, richSummaryIntro, useCasesSummary) — convert to Lexical format if JSON uses plain text.

### R4 — Create/update domain sets
For each domain that belongs to a set:
- If the set already exists in seed, add the domain as a new member

### R5 — Handle overlaps
For domains that appear in both A1 seed and Codex output:
- Keep A1 content as primary

- Do not duplicate the domain entry
- Report which domains overlapped and how they were merged

## Risks / Pitfalls

| Risk | Mitigation |
|------|------------|
| **Codex content has field format mismatches** | Validation at Checkpoint A catches these. Rich text fields may need Lexical JSON conversion. Fix format, not content. |

| **Set policy rules** | IDN+romaji pairs must be bundle_only. Individual domains without a matching pair use allow_individual. Follow existing set patterns from A1. |
