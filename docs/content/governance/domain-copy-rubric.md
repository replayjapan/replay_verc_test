# Domain Copy Rubric — rePlay Domains

## Goal

Describe each domain as a strategic brand or business opportunity in a way that feels plausible, premium, and useful to a serious buyer.

## Description Length Targets

| Field | Target Length |
|---|---|
| Short description | 120–220 Japanese characters |
| Detailed description / body | 220–450 Japanese characters |
| searchExcerpt | 80 double-byte characters max (hard limit) |

Keep it dense, not bloated.

## Required Content Moves

Each domain description should cover most of these naturally:

1. What kind of business, service, media, or platform the domain could support
2. Who the likely audience or buyer is
3. Why the name has strength in the Japanese market
4. What makes this domain distinct from a weaker generic alternative
5. Where relevant, how it fits within a broader set narrative

## Valid Use Cases

**Good use cases are:**
- Plausible
- Commercially believable
- Suited to the exact domain string
- Relevant to Japan or a clear bilingual/global use case

**Bad use cases are:**
- Too broad to mean anything
- Too speculative
- Unrelated to the domain name
- Stuffed in just to create length

## Allowed Speculation

**Allowed:**
- "could become"
- "well suited for"
- "works naturally for"
- "fits brands targeting…"

**Not allowed:**
- Implying current customers, traffic, partners, investor interest, search demand, or operational status unless Developer explicitly provided it

## Set Narrative vs Individual Positioning

**For set members:**
- Keep one shared thesis line in mind (see set-thesis-lines.md)
- Still give each domain its own role

**Rule:**
- Set thesis explains the ecosystem
- Individual copy explains the specific job of that domain inside the ecosystem
- Do NOT duplicate nearly identical copy across set members

## Pricing Effect on Tone

Pricing affects restraint, not bragging.

| Tier | Tone |
|---|---|
| Higher-threshold domains | More selective, foundational, brand-defining |
| Lower-threshold but strong | Focused and practical |

- Never mention internal tier labels (Trophy, Premium, Niche, etc.) on the frontend
- Never let price logic turn into domainer chest-thumping

## Pricing Tiers (Internal Reference Only)

| Tier | Examples | Minimum Offer |
|---|---|---|
| Trophy | アニメ.com, 派遣.com | ¥50,000,000+ |
| Premium | ホテルズ.com, セール.com, diamonds.jp | ¥5,000,000 - ¥20,000,000 |
| Geographic (set-only) | Boston Set, LA Set, Honolulu Set | ¥10,000,000+ per set |
| Niche | sneakers.jp, princess.jp, salsa.jp | ¥1,000,000 - ¥5,000,000 |
| Set members | 海外ホテル.com, ハワイホテル.com | ¥500,000 - ¥3,000,000 |

## Set-Only Sales Rule

Any set where IDN and romaji domains match must be sold together. Example: boston.jp + ボストン.jp cannot be sold individually — the full set starts at ¥10,000,000 minimum. Domains without an IDN/romaji pair (e.g., sneakers.jp, princess.jp) can be sold individually.

## Prohibited Claims

Do not claim or imply:
- Active business operations
- Current traffic
- Current leads
- Buyer demand
- Verified search volume
- Ranking advantages
- Guaranteed SEO performance
- Exclusivity beyond what the name itself naturally implies
- Legal/trademark safety unless separately verified

## Distinctiveness Rule

Every domain entry must have:
1. One specific commercial angle
2. One specific audience
3. One specific reason the name matters

**Test:** If three descriptions could swap domains and still "work," they are too generic.

## Search Keywords Rule

searchKeywords are functional fields, not prose.

**Use:**
- Clean, relevant keyword variants
- Japanese-first terms
- A few English terms only where genuinely useful

**Do not** turn them into spam lists.
